🔧 tech deep-dive

What can you build with zero dependencies?

I challenged myself to build a complete algorithmic trading training simulator using nothing but Python's standard library. No FastAPI. No Flask. No NumPy. No Pandas. Not even a requirements.txt file.

The result is NovaCap · Principal Strategist — and here's what Python's standard library alone delivered:

  • http.server + socketserver → A threaded REST mock exchange with live order books
  • sqlite3 → Full double-entry accounting engine with P&L tracking and audit trails
  • ast → Static code auditor that parses your strategy source and flags 9 categories of violations
  • threading + subprocess → Concurrent exchange, dashboard, and test execution
  • cmd.Cmd → Full CLI with 10 commands: start, run, eod, advance, audit, and more
  • pathlib + json → Workspace generation, state persistence, market data serialization

No microservices. No containers. No docker-compose. One Python process. Three servers. Full simulation.

I think there's something poetic about building complex systems with simple tools. It forces you to understand every layer of abstraction — and it keeps your deploy surface area vanishingly small.

How I built a code auditor using Python's built-in AST module

In NovaCap, every piece of strategy code that a trainee writes gets parsed and inspected before it ever executes.

Python's ast module gives you the ability to walk through any Python source file's syntax tree — and with that, you can enforce code quality rules automatically.

My AST auditor checks for 9 violation categories:

  • Critical: Missing error handling, eval()/exec() calls, mutable default arguments
  • Structural: Try blocks without except, bare except clauses, function complexity beyond nesting limits
  • Safety: Global state mutations, unsafe input handling, missing return annotations on critical functions
  • Style: Deeply nested loops (beyond 3 levels), functions over 50 lines, excessive branching

Each violation carries a penalty. Two critical violations = automatic 0 for the day. The scoring system is designed to be harsh because the stakes in real trading are even harsher.

The entire auditor is about 300 lines of Python — a testament to how powerful the standard library really is.

If you haven't explored Python's ast module, I highly recommend it. It's one of those tools that changes how you think about code analysis and meta-programming.

🏗️ architecture

The architecture of a single-process trading simulator

One of the most interesting engineering challenges in NovaCap was getting a complete trading simulation — exchange, dashboard, CLI, ledger, and evaluation engine — to run inside a single Python process.

Here's how it breaks down at runtime:

arbitrage_academy.py (3,961 lines) starts one threading.Thread for the mock exchange HTTP server on port 8080, and another for the web dashboard on port 8081. Meanwhile, the main thread runs the cmd.Cmd CLI loop. All three communicate through a shared SimulationState object protected by locks.

Thread 1 — Mock Exchange: Serves REST endpoints for order books, tickers, and trade execution. Maintains synthetic market state with configurable volatility.

Thread 2 — Web Dashboard: Serves a dark-themed SPA-style UI with live ticker, code editor, evaluation history, and ledger explorer.

Main Thread — CLI: The Principal Strategist persona. Accepts 10 commands and orchestrates the full training workflow.

Persistent Layer — SQLite: Double-entry ledger with 9 standard accounts. Full transaction log. Balances must always reconcile.

No Docker. No microservices. No dependency injection framework. Just threads, locks, and careful state management.

Sometimes the simplest architecture is the right one.

⚡ performance

Zero dependencies means zero deployment friction.

I keep seeing projects with 500+ npm packages or 50+ pip dependencies for relatively simple applications. Each one is a liability: a potential security vulnerability, a license conflict, a future maintenance headache.

For NovaCap, I made a deliberate decision: no external dependencies.

This has concrete benefits:

  • Clone and run in 5 seconds. No pip install. No failed builds. No version conflicts.
  • No attack surface. Standard library modules are maintained by the PSF and are part of every Python installation.
  • Deterministic behavior. Your code runs identically on every machine because there's no dependency version to drift.
  • Forced simplicity. When you can't reach for NumPy, you write your own moving average. And you understand it deeply.

The entire simulator is about 8,000 lines across three files. The core engine (arbitrage_academy.py) is a single file of 3,961 lines — all standard library.

Before you add that next dependency to your project, ask yourself: can I do this with what I already have?

You might surprise yourself with what the standard library can do.

📊 accounting

If your debits don't equal your credits, you fail the day.

In NovaCap's training simulator, every trade your algorithm makes is logged to a double-entry SQLite ledger. If your positions cause an imbalance — even by a single satoshi — the evaluation engine flags it and deducts 30 points from your score.

This is not arbitrary cruelty. This is how real financial systems work.

The ledger uses 9 standard accounts:

  • Asset accounts: Cash, BTC, ETH, USDT
  • Liability accounts: Margin, Pending Settlements
  • Equity: Opening Balance, Retained Earnings
  • Income/Expense: Realized P&L, Unrealized P&L

Every transaction records: timestamp, symbol, quantity, price, debit account, credit account, and a unique audit ID. The entire history is queryable via the web dashboard.

Why go this far for a training tool? Because financial software has no margin for error. Learning to think in terms of accounting integrity — debits always equal credits, every position is traceable — is one of the most important skills for any quantitative developer.

Could I have packaged this as a paid course? Probably. Could I have put it behind a paywall? Definitely.

The stack: Python 3.11+ (standard library only)
The constraint: Zero external dependencies

🎯 evaluation

How do you score a trading strategy when you don't have a crystal ball?

In NovaCap, the answer is: you don't score the strategy's P&L. You score the process.

The End-of-Day evaluation pipeline runs three independent checks, each with a specific weight:

Stage 1 — AST Audit (max 40 points)
Your source code is parsed into an abstract syntax tree. The auditor walks the tree looking for 9 categories of violations. Critical issues (missing exception handling, unsafe eval calls) cost 40 points each. Warnings (excessive nesting, missing annotations) cost 10 each.

Stage 2 — Test Suite (max 30 points)
Each day comes with a hidden test suite. Your strategy code is executed against it. Every failing test costs 30 points.

Stage 3 — Ledger Integrity (max 30 points)
Your trading activity is loaded from SQLite. Every transaction is checked: debits = credits, no orphan entries, complete audit trail. Any imbalance costs 30 points.

Result: a score of 0–100.
Below 100? You don't advance. You refactor, you fix, you retry.

The philosophy: in real trading, you can't control market outcomes. But you can control code quality, test coverage, and accounting integrity. That's what NovaCap measures. And that's what matters.
